# Tool Calling / Function Calling

Endpoint:

```http
POST /tool-assistant
```

The model receives the user message plus a list of backend tools. It can choose
one or more tools, the backend validates and runs them, then the tool results
are returned to the model so it can write the final answer.

The production-style pieces are:

- **Tool registry:** every tool has a schema, risk level, and approval policy.
- **Tool runner:** one place validates arguments, checks approval, executes the
  handler, limits result size, and emits audit logs.
- **Tool metadata:** every API response shows tool risk, approval status,
  execution status, and errors.

---

## Available tools

### `get_weather`

Returns a deterministic mock weather report for a location.

Risk: `read_only`

Arguments:

```json
{
  "location": "Yerevan",
  "unit": "celsius"
}
```

### `search_docs`

Runs semantic search over uploaded documents using the existing documents
pipeline.

Risk: `read_only`

Arguments:

```json
{
  "query": "retry logic",
  "top_k": 3,
  "filename": "optional-file.txt",
  "keyword": "optional keyword",
  "min_similarity": 0.2
}
```

### `create_ticket`

Creates a ticket draft only. It does not send anything to an external ticketing
system.

Risk: `draft`

Arguments:

```json
{
  "title": "Improve retry documentation",
  "description": "Add examples for timeout and rate-limit retries.",
  "priority": "high"
}
```

### `send_email_draft`

Creates an email draft only. It does not send email.

Risk: `draft`

Arguments:

```json
{
  "to": "teammate@example.com",
  "subject": "Draft subject",
  "body": "Draft body"
}
```

## Request example

```bash
curl -X POST "http://127.0.0.1:8000/tool-assistant" \
  -H "Content-Type: application/json" \
  -d '{"message":"Search my docs for retry logic and create a high-priority ticket draft to improve the docs."}'
```

## Safety behavior

- Tool arguments are validated with Pydantic before execution.
- Unknown tools return an error result instead of being executed.
- Tool risk is explicit: `read_only`, `draft`, `write`, or `dangerous`.
- Tools that require approval are blocked by the runner until approved.
- Tool results are size-limited before being returned to the model.
- Every execution attempt writes a structured audit log.
- `create_ticket` creates only a draft and marks it as requiring human
  confirmation.
- `send_email_draft` creates only a draft, sets `"sent": false`, and marks it
  as requiring human confirmation.
- `search_docs` handles an empty vector store as a tool result, so the assistant
  can explain that documents must be uploaded first.
