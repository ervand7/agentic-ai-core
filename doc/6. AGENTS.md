# Agents

Endpoint:

```http
POST /research-agent
```

Given a goal, it runs a bounded
loop of `plan -> reason -> act -> observe`, decides for itself how many tool
steps to take, optionally critiques its own draft, and returns a final report
together with a full, auditable trace.

---

## Chatbot vs. agent

| | Chatbot (`/ask`) | Tool assistant (`/tool-assistant`) | Agent (`/research-agent`) |
| --- | --- | --- | --- |
| Turns | one | exactly one tool round | a loop, length chosen by the model |
| Decides #steps | no | no (fixed: tools â†’ answer) | **yes** |
| State | none | implicit | explicit `AgentState` + budgets |
| Stops because | reply is written | one round done | model stops calling tools, or a budget is hit |

The conceptual leap is the **loop plus a budget**. The model â€” not the code â€”
decides when it has enough evidence; the code only enforces hard safety limits.

---

## The agent loop

```
plan  ->  [ reason -> act (tools) -> observe ]*  ->  (force final if budget-cut)  ->  reflect
```

1. **Plan.** A structured-output call asks the model for an ordered list of
   research steps (`ResearchPlan`). This is the "planning" concept.
2. **Loop.** Each iteration calls the model with the tool set. If it returns
   tool calls, every call is executed through the existing `ToolRunner`
   (validation, risk, approval, audit), the results are appended to the
   conversation, and the loop continues. If it returns **no** tool calls, the
   model is signalling it is done â€” that is the natural termination.
3. **Force final.** If a budget cut ends the loop before the model finished, a
   single no-tools call synthesizes a best-effort report from what was gathered.
4. **Reflect.** An optional critic call (`AgentCritique`) judges the draft and
   may return a revised version. This is the "reflection" concept.

The conversation history (the growing `messages` list) is the agent's
short-term **memory**; `AgentState` is its explicit **state**.

---

## Safety budgets (agent failure modes)

Letting a model loop is exactly what causes the classic agent failure modes, so
each one is guarded explicitly:

| Failure mode | Guard |
| --- | --- |
| Infinite loop | `AGENT_MAX_ITERATIONS` hard cap |
| Runaway cost / expensive calls | `AGENT_MAX_TOTAL_TOKENS` budget, summed across **every** call (plan + each step + final + reflection) |
| Wrong tool selection | focused tool set (`search_docs`, `create_ticket`), low temperature, `ToolRunner` validation |
| Bad reasoning | optional reflection/critic pass |
| Unsafe actions | reused tool policy: drafts only, risk levels, audit logs |

`AgentStopReason` records *why* a run ended: `completed`, `max_iterations`, or
`token_budget`.

## Request example

```bash
curl -X POST "http://127.0.0.1:8000/research-agent" \
  -H "Content-Type: application/json" \
  -d '{"topic":"How does this project handle retries and timeouts?"}'
```

Optional `max_iterations` (1â€“10) overrides the loop cap for a single run:

```bash
curl -X POST "http://127.0.0.1:8000/research-agent" \
  -H "Content-Type: application/json" \
  -d '{"topic":"Summarize the RAG pipeline","max_iterations":4}'
```

> The agent's main evidence tool is `search_docs`, so upload documents first
> (`POST /documents/upload`) for grounded reports.

## Configuration

| Variable | Purpose | Default |
| --- | --- | --- |
| `AGENT_MAX_ITERATIONS` | Hard cap on loop steps | `6` |
| `AGENT_MAX_TOTAL_TOKENS` | Token budget across the whole run | `12000` |
| `AGENT_TEMPERATURE` | Sampling temperature for agent calls | `0.1` |
| `AGENT_REPORT_MAX_TOKENS` | Max tokens for a forced final report | `800` |
| `AGENT_ENABLE_REFLECTION` | Run the self-critique pass | `true` |
| `PROMPT_RESEARCH_AGENT_PLANNER_SYSTEM` | Planner system prompt | sensible default |
| `PROMPT_RESEARCH_AGENT_SYSTEM` | Agent loop system prompt | sensible default |
| `PROMPT_RESEARCH_AGENT_CRITIC_SYSTEM` | Critic system prompt | sensible default |

---

## When NOT to use an agent

If the number and order of steps are known ahead of time (like the RAG flow:
retrieve â†’ answer), a fixed pipeline is cheaper, faster, and more reliable than
an agent. Reach for an agent only when the steps genuinely cannot be planned in
advance.
