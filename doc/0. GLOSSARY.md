# AI Engineering Glossary

A quick reference for the AI engineering terms used in this project. Each entry
gives a plain-language definition and points to where the concept shows up in
the codebase.

---

## Models & generation

### LLM (Large Language Model)
A model trained on large amounts of text that predicts the next token to
generate human-like responses. Here it powers `/ask`, `/classify`,
`/summarize`, etc. via OpenAI.
*Code:* `app/domains/ai_tasks/infrastructure/openai_llm_provider.py`

### Model
The specific LLM used to answer requests. Default is `gpt-4o-mini`.
*Config:* `OPENAI_MODEL`

### Temperature
A number (`0.0`–`2.0`) that controls randomness/creativity of the output.
Low (e.g. `0.3`) → focused, deterministic answers; high → more varied,
creative answers. We default to `0.3` for predictable results.
*Config:* `OPENAI_TEMPERATURE`

### Token
The unit a model reads and writes — roughly a word piece (a few characters).
Both input and output are measured in tokens, and usage is logged per request.

### Max tokens
The cap on how many tokens the model may generate in its response. Limits
length and cost. Default `300`.
*Config:* `OPENAI_MAX_TOKENS`

### Prompt
The text instructions sent to the model. A **system prompt** sets the
assistant's role/behavior; the **user prompt** is the actual input.
*Config:* the `PROMPT_*_SYSTEM` settings.

### Prompt template / prompt versioning
Reusable, named prompts with an attached version (e.g. `ask_v1`). Versioning
lets you change wording deliberately and track which prompt produced a result.
*Code:* `app/domains/ai_tasks/domain/prompts.py`

### Streaming
Sending the answer token-by-token as it is generated, instead of waiting for
the full response. Powers `/ask-stream` and the live UI typing effect.
*Code:* `OpenAILLMProvider.stream`

### Structured output / JSON schema
Forcing the model to return JSON that matches an exact shape (a schema),
instead of free text. Used for classification, keyword extraction, and text
analysis so the output is machine-readable. `strict: True` enforces the schema.
*Code:* `app/domains/ai_tasks/domain/response_formats.py`

## Embeddings & semantic search

### Embedding
A piece of text converted into a list of numbers (a **vector**) that captures
its meaning. Texts with similar meaning get similar vectors. We embed each
document chunk and each search query.
*Config:* `OPENAI_EMBEDDING_MODEL` (default `text-embedding-3-small`, ~1536 numbers per vector)

### Vector / vector space
The list of numbers produced by an embedding. You can imagine each vector as a
point/direction in a high-dimensional "meaning space"; nearby vectors mean
similar content.

### Cosine similarity
A score from `0` (unrelated) to `1` (nearly identical) that measures how close
two vectors point in the same direction. It's how we rank search results by
meaning.
*Code:* `app/domains/documents/domain/chunking.py` → `cosine_similarity`

### Semantic search
Searching by meaning rather than exact keywords: embed the query, then find the
chunks whose vectors are most similar.
*Endpoint:* `POST /documents/search`

### Chunking
Splitting a long document into smaller pieces before embedding, because models
and search work better on focused, bite-sized text.
*Config:* `DOCUMENT_CHUNK_SIZE`

### Chunk overlap
Repeating a small tail of one chunk at the start of the next, so meaning that
sits on a boundary isn't lost between chunks.
*Config:* `DOCUMENT_CHUNK_OVERLAP`

### Hybrid search
Combining meaning-based (semantic) matching with a literal keyword check, then
blending both signals into the final ranking (here: 80% meaning + 20% keyword).
*Field:* `keyword` on `/documents/search`

### top_k
How many of the best-matching results to return. Default `3`.

### Similarity threshold (`min_similarity`)
A minimum cosine-similarity score; matches below it are dropped to hide weak or
unrelated results.

## RAG (Retrieval-Augmented Generation)

### RAG
Answering a question by first **retrieving** relevant text from your own
documents, then passing it to the LLM as context so the **generated** answer is
grounded in that text rather than the model's memory.
*Endpoint:* `POST /documents/ask` · *Code:* `AnswerQuestionService`

### Retrieval
The "R" in RAG: embedding the question and pulling the most similar chunks from
the vector store to use as context.
*Code:* `app/domains/documents/infrastructure/qdrant_vector_store.py` → `search`

### Document loading
Turning an uploaded file (`.txt`, `.pdf`) into plain text before chunking.
*Code:* `app/domains/documents/infrastructure/document_loader.py`

### Context injection (context stuffing)
Inserting the retrieved chunks into the prompt (as a numbered context block) so
the model can answer from them.
*Code:* `app/domains/documents/domain/rag.py` → `build_user_prompt`

### Grounding / faithfulness
How strongly the answer is supported by the provided context. A grounded answer
makes no claims that aren't backed by a retrieved chunk.

### Citation / source attribution
Returning *which* chunks (file + snippet) an answer is based on, and asking the
model to reference them inline as `[1]`, `[2]`. Lets users verify the answer.
*Code:* `app/domains/documents/domain/rag.py` → `build_citations`

### Hallucination
When an LLM states something confidently that isn't true or isn't supported by
the context. RAG + citations + low temperature reduce it.

### Abstention ("I don't know")
Choosing not to answer when retrieval finds nothing relevant, instead of
guessing. Here it returns a canned message with `used_context: false` and skips
the LLM call entirely.
*Code:* `AnswerQuestionService` + `rag.NO_CONTEXT_ANSWER`

### Lost in the middle
A failure mode where stuffing too much context buries the relevant line and the
model misses it. Mitigated by a modest `RAG_TOP_K` and focused chunks.

## Vector database (Qdrant)

### Vector database
A database built to store vectors and find the most similar ones quickly. We
use **Qdrant**.
*Config:* `QDRANT_URL`

### Collection
The container that holds all the points (vectors + data), similar to a table in
a normal database.
*Config:* `QDRANT_COLLECTION_NAME`

### Point
One stored record: a unique `id`, its `vector` (the embedding), and a `payload`
(extra data like the original `text` and `filename`).

### Payload
The non-vector data attached to a point — here the source chunk text and its
filename, used for returning results and filtering.

### Index (HNSW)
A data structure that makes "find the nearest vectors" fast without scanning
everything. Qdrant uses an HNSW vector index; we also add a `filename` index for
fast filtering.

## Reliability & operations

### Retries with exponential backoff
Automatically re-sending a failed request after waiting, with the wait time
growing each attempt. Smooths over transient provider errors.
*Config:* `OPENAI_MAX_RETRIES`, `OPENAI_RETRY_BASE_DELAY_SECONDS`

### Timeout
The maximum time to wait for the provider before giving up on a request.
*Config:* `OPENAI_TIMEOUT_SECONDS`

### Latency
How long a request takes, in milliseconds. Logged per request to monitor
performance.

### Token usage
The number of tokens consumed by a request (input + output). Logged for cost and
performance tracking.

### Request ID
A unique identifier attached to each request (returned as `x-request-id`) so
logs for one call can be traced end to end.

### NLP (Natural Language Processing)
The broad field of getting computers to understand and work with human
language — the umbrella over summarization, sentiment, keyword extraction, and
translation tasks in this project.
