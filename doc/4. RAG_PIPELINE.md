# RAG Pipeline — Chat with Your Documents

**RAG = Retrieval-Augmented Generation**: first *retrieve* relevant knowledge
from your own documents, then *augment* the LLM prompt with it so the model
*generates* an answer grounded in that knowledge (instead of guessing from
its training data).

This project exposes RAG through one endpoint:

```http
POST /documents/ask
```

It builds on the existing upload/search pipeline. If you haven't read it yet,
see `doc/1. DOCUMENTS_SEMANTIC_SEARCH.md` for how upload + embeddings + Qdrant
work — this doc focuses on the *generation* half.

---

## The full pipeline

```
            ┌─────────────────────── INGEST (once, per upload) ──────────────────────┐
 file ─▶ load text ─▶ split into chunks ─▶ embed each chunk ─▶ store vectors in Qdrant
            └────────────────────────────────────────────────────────────────────────┘

            ┌────────────────────────── ASK (per question) ─────────────────────────┐
 question ─▶ embed ─▶ retrieve top-k chunks ─▶ inject as context ─▶ LLM ─▶ answer + citations
            └────────────────────────────────────────────────────────────────────────┘
```

Mapped to the learning checklist:

| Step | Where it lives |
|---|---|
| 1. Basic RAG pipeline | `AnswerQuestionService.execute` |
| 2. Document loading | `infrastructure/document_loader.py` (`.txt`, `.pdf`) |
| 3. Text splitting | `domain/chunking.py` → `chunk_text` |
| 4. Embedding generation | `infrastructure/openai_embedding_provider.py` |
| 5. Retrieval | `infrastructure/qdrant_vector_store.py` → `search` |
| 6. Context injection | `domain/rag.py` → `build_user_prompt` |
| 7. Answer generation | `infrastructure/openai_answer_generator.py` |
| 8. Source citations | `domain/rag.py` → `build_citations` |
| 9. "I don't know" behavior | `AnswerQuestionService` + `NO_CONTEXT_ANSWER` |

## Step by step

### 1. Document loading

`document_loader.extract_text()` turns uploaded bytes into plain text.

- `.txt` → decoded as UTF-8.
- `.pdf` → parsed with `pypdf` (text-based PDFs only; scanned/image PDFs need
  OCR, which we don't do).

Only clean text moves forward, so the rest of the pipeline never cares about
the original file format.

### 2. Text splitting (chunking)

Long text is split into overlapping chunks (`DOCUMENT_CHUNK_SIZE`,
`DOCUMENT_CHUNK_OVERLAP`). Chunks are the unit we embed and later retrieve.

### 3–5. Embed + store + retrieve

Each chunk is embedded and stored in Qdrant as a point (`vector` + `payload`
with `text` and `filename`). At ask-time the **question** is embedded the same
way and Qdrant returns the nearest chunks by cosine similarity. See
`doc/2. VECTOR_DATABASE_USAGE.md`.

### 6. Context injection

The retrieved chunks are formatted into a numbered context block and placed in
the user message (`domain/rag.py`):

```
Context snippets:
[1] (source: handbook.txt)
Retries use exponential backoff...

[2] (source: handbook.txt)
Timeouts return a friendly message...

Question: How does retry logic work?

Answer using only the context above and cite the snippet numbers you use.
```

The **system prompt** (`PROMPT_RAG_SYSTEM`) carries the rules: answer only from
context, cite snippets like `[1]`, and say "I don't know" if the answer isn't
there.

### 7. Answer generation

`OpenAIAnswerGenerator` sends `system + user` messages to the chat model at a
low temperature (`RAG_TEMPERATURE`, default `0.1`) so the answer stays grounded.

### 8. Source citations

Every retrieved chunk is returned as a citation (`index`, `filename`, `text`,
`similarity`). The model is asked to reference them inline as `[1]`, `[2]`, and
the API also returns the full citation list so a UI can show its sources.

### 9. "I don't know" behavior (abstention)

Two layers protect against made-up answers:

1. **Retrieval guard** — if nothing clears `RAG_MIN_SIMILARITY`, the service
   returns a canned "I don't know" (`used_context: false`) **without calling the
   LLM**. No context ⇒ no point paying for a hallucination.
2. **Prompt guard** — even with context, the system prompt tells the model to
   admit when the answer isn't supported.

## Request / response

**Request:**

```json
{
  "question": "How does retry logic work?",
  "top_k": 4,
  "filename": "handbook.txt",
  "min_similarity": 0.2
}
```

`top_k`, `filename`, and `min_similarity` are optional (defaults come from the
`RAG_*` settings).

**Response (grounded):**

```json
{
  "question": "How does retry logic work?",
  "answer": "Retries use exponential backoff [1], and timeouts return a friendly message [2].",
  "used_context": true,
  "citations": [
    { "index": 1, "filename": "handbook.txt", "text": "Retries use exponential backoff...", "similarity": 0.91 },
    { "index": 2, "filename": "handbook.txt", "text": "Timeouts return a friendly message...", "similarity": 0.86 }
  ],
  "model": "gpt-4o-mini",
  "tokens_used": 320
}
```

**Response (no relevant context):**

```json
{
  "question": "What is the capital of France?",
  "answer": "I don't know based on the uploaded documents. I couldn't find anything relevant to that question.",
  "used_context": false,
  "citations": [],
  "model": null,
  "tokens_used": 0
}
```

## Configuration

| Setting | Default | Purpose |
|---|---|---|
| `RAG_TOP_K` | `4` | How many chunks to retrieve as context. |
| `RAG_MIN_SIMILARITY` | `0.2` | Drop weakly-related chunks (guards retrieval quality). |
| `RAG_TEMPERATURE` | `0.1` | Low → grounded, less creative answers. |
| `RAG_MAX_TOKENS` | `500` | Max length of the generated answer. |
| `PROMPT_RAG_SYSTEM` | see `config.py` | Behavior rules (cite, abstain, no invention). |

## Common RAG bugs (and how this project mitigates them)

### Bad chunking
Chunks too large blur meaning; too small lose context. Symptoms: vague answers,
citations that don't quite contain the fact.
→ Tune `DOCUMENT_CHUNK_SIZE` / `DOCUMENT_CHUNK_OVERLAP`. Overlap keeps facts that
straddle a boundary intact.

### Irrelevant retrieval
The model is handed off-topic chunks and answers confidently from noise.
→ `RAG_MIN_SIMILARITY` filters weak matches; `filename` narrows the search; the
prompt forbids using outside knowledge.

### Too much context
Stuffing many/long chunks raises cost and latency and can *bury* the relevant
line ("lost in the middle").
→ Keep `RAG_TOP_K` modest and chunks focused; raise `min_similarity` to admit
only the strongest matches.

### Missing citations
An answer with no sources can't be trusted or verified.
→ Citations are returned for every retrieved chunk, the prompt requires inline
`[n]` references, and the UI renders them under the answer.
