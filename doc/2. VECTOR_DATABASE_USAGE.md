# Vector Database Usage

This file explains, in plain words, the extra search features we added on top of Qdrant.

Everything here is used through one endpoint:

```http
POST /documents/search
```

You always send a `query`. The other fields are optional.

---

## The search fields

| Field | Required? | What it does |
|---|---|---|
| `query` | yes | Your question in normal language. |
| `top_k` | no (default 3) | How many results to return. |
| `filename` | no | Only search inside one file. |
| `keyword` | no | Require a specific word to appear. |
| `min_similarity` | no | Hide weak/unrelated matches. |

---

## How the data is saved in the database

When you upload a file, we split it into small pieces ("chunks"). Each chunk is saved
in Qdrant as one **point**. A point has three parts:

- `id` — a unique id for the chunk (a random UUID).
- `vector` — the chunk's meaning turned into a list of numbers (the embedding).
- `payload` — extra info we attach: the original `text` and the source `filename`.

**Example:** uploading `coffee.txt` with two chunks creates two points like this:

```json
[
  {
    "id": "1f3a9c2e8b7d4f10a1c2b3d4e5f60718",
    "vector": [0.0123, -0.0456, 0.0789, "... ~1536 numbers total ..."],
    "payload": {
      "text": "To clean the milk frother, remove the nozzle and rinse it.",
      "filename": "coffee.txt"
    }
  },
  {
    "id": "9b8a7c6d5e4f3021b1a2c3d4e5f60912",
    "vector": [0.0222, 0.0011, -0.0307, "... ~1536 numbers total ..."],
    "payload": {
      "text": "To brew espresso, press the button twice.",
      "filename": "coffee.txt"
    }
  }
]
```

What each part is used for:

- The `vector` is what search compares against (the "meaning").
- The `payload.text` is what we return to you in results.
- The `payload.filename` is what the `filename` filter uses.
- All points live inside one **collection** (think of it like a table).

---

## 1) Indexes

We set this up automatically when the collection is created.
Two things happen behind the scenes:

- A vector index (HNSW) → makes "find similar meaning" fast.
- A `filename` index → makes "filter by file" fast.

You don't need to configure this. It just works.

---

## 2) Filter by file (`filename`)

Use this when you only care about one document.

**Example:** search only inside `coffee.txt`:

```bash
curl -X POST "http://localhost:8000/documents/search" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "how do I clean it?",
    "filename": "coffee.txt"
  }'
```

Result: only chunks that came from `coffee.txt` are searched. Everything from other
files is ignored.

---

## 3) Keyword + meaning together (`keyword`)

Normal search matches by **meaning**. Sometimes you also want to force a specific
word to be present. That's what `keyword` does (this is called "hybrid search").

**Example:** find chunks about cleaning, but only ones that mention "frother":

```bash
curl -X POST "http://localhost:8000/documents/search" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "how do I clean it?",
    "keyword": "frother"
  }'
```

How it works in simple terms:

- The database finds chunks with similar meaning.
- We then keep only the ones that actually contain your keyword.
- Final ranking mixes both signals: **80% meaning + 20% keyword**.

So a chunk that matches the meaning *and* contains the word ranks highest.

---

## 4) Hide weak matches (`min_similarity`)

Similarity is a score from `0` (unrelated) to `1` (almost identical).
Use `min_similarity` to drop anything below a score you choose.

**Example:** only return strong matches (score `0.8` or higher):

```bash
curl -X POST "http://localhost:8000/documents/search" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "how do I clean the milk frother?",
    "min_similarity": 0.8
  }'
```

- High value (e.g. `0.9`) → fewer, very precise results.
- Low value (e.g. `0.2`) → more results, but some may be loosely related.

If everything gets filtered out, your threshold is probably too high.

---

## 5) Everything combined

You can mix all of them in one request.

**Example:** in `coffee.txt`, find strong matches about cleaning that mention "nozzle",
and return the top 2:

```bash
curl -X POST "http://localhost:8000/documents/search" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "how do I clean the machine?",
    "filename": "coffee.txt",
    "keyword": "nozzle",
    "min_similarity": 0.5,
    "top_k": 2
  }'
```

---

## Things to keep in mind (limitations)

- **Keyword match is basic.** It checks whether the word is present, not clever
  language ranking (no BM25-style scoring).
- **`filename` is exact-match only.** No partial names or wildcards yet.
- **A high `min_similarity` can hide good results.** Start low and raise it slowly.
- **Defaults are fine for small data.** Heavy production use would need extra tuning.
